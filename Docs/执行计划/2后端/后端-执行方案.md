## M0 — 项目骨架 + 认证（P0-1）

### 目标
- Laravel 项目跑起来
- JWT 认证打通
- 4 个认证接口可用

### 任务清单

#### 0.1 项目初始化
- [ ] 确认 `app/` 目录是 Laravel 13 工程（不是别的项目）
- [ ] `composer install`
- [ ] `.env` 配置：
  - `DB_CONNECTION=mysql`
  - `DB_DATABASE=article_platform`
  - `JWT_SECRET=...`（已有，确认在）
  - `JWT_TTL=60`
- [ ] `php artisan key:generate`
- [ ] `php artisan migrate`（确保空库能起）

#### 0.2 安装 JWT 认证
- [ ] 确认 `tymon/jwt-auth` 已装（见 `插件.md`）
- [ ] `config/auth.php` 把 `defaults.guard` 改为 `api`，`api` driver 改为 `jwt`
- [ ] `App\Models\User` 实现 `Tymon\JWTAuth\Contracts\JWTSubject` 接口（getJWTIdentifier, getJWTCustomClaims）
- [ ] `database/migrations/xxxx_create_users_table.php` 加字段：
  - `id` 用 ULID（26 位字符串，HasUlids trait，与接口文档和 M1+ 一致）
  - `name` string(50) **唯一**（接口文档未列但 DB 设计文档 3.1 要求）
  - `role` enum('reader','author','admin') default 'reader'
  - `avatar` string nullable
  - `bio` text nullable
  - `coin_balance` int default 0
  - `gold_balance` decimal(10,2) default 0
  - `last_login_at` timestamp nullable
  - `last_coin_at` timestamp nullable（每日硬币）
  - `email_verified_at` timestamp nullable（加字段，M0 不写验证流程）
  - `remember_token` string(100) nullable
  - `deleted_at` softDeletes
- [ ] User model `$fillable` / `$hidden` / `$casts` 同步更新

#### 0.3 实现认证接口

**文件**：`app/Http/Controllers/Api/AuthController.php`

| 方法 | 路由 | 说明 |
|---|---|---|
| `register` | `POST /api/auth/register` | 校验 + 创建用户 + 返回 token |
| `login` | `POST /api/auth/login` | 校验 + 返回 token + 更新 `last_login_at` |
| `logout` | `POST /api/auth/logout` | `auth:api` 中间件 + `JWTAuth::invalidate()` |
| `me` | `GET /api/auth/me` | `auth:api` 中间件 + 返回当前用户 |

**关键实现**：
- `register` 用 `Validator::make`，`password_confirmation` 校验
- `login` 用 `Auth::attempt(['email' => $email, 'password' => $password])`
- `me` 用 `Auth::user()`，按接口文档返回字段（id/name/email/role/avatar/bio/coin_balance/gold_balance/created_at）
- 错误统一抛 `ValidationException` 或返回 `{ "message": "...", "errors": {...} }`

#### 0.4 配置 CORS
- [ ] `config/cors.php` 设置 `allowed_origins` 包含前端开发地址：
  - `http://localhost:5173`（Vite 默认）
  - `http://localhost:8080`（UniApp H5 常用）
  - `http://localhost:9000`（UniApp H5 备用）
- [ ] `supports_credentials => true`

#### 0.5 创建管理员 Seeder
- [ ] `database/seeders/AdminSeeder.php`：创建 `admin@example.com` / `password`，role=admin
- [ ] `DatabaseSeeder` 调用

### 验证方式

```bash
# 注册
curl -X POST http://127.0.0.1:8000/api/auth/register \
  -H "Content-Type: application/json" \
  -d '{"name":"测试","email":"test@example.com","password":"password123","password_confirmation":"password123"}'
# 期望：201 + token

# 登录
curl -X POST http://127.0.0.1:8000/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"test@example.com","password":"password123"}'
# 期望：200 + token

# 当前用户
curl http://127.0.0.1:8000/api/auth/me \
  -H "Authorization: Bearer <token>"
# 期望：200 + 用户对象
```

跑通后再开前端对话。

## M1 — 文章 CRUD（P0-2）

### 目标
- 文章表 + 标签 + 分类 + 快照（先建表，暂不存）
- 6 个接口可用
- Markdown 原文存 + 服务端渲染 HTML 一起返回

### 任务清单

#### 1.1 数据库迁移

**`articles` 表**（按 `数据库设计.md`）：
- `id` ulid 主键
- `user_id` ulid 外键
- `category_id` ulid nullable
- `status` enum('draft','pending','first_pass','published','first_reject','modify_required','appealing','second_pass','second_reject') default 'draft'
- `title` string(200)
- `slug` string unique
- `content` longText（Markdown 原文）
- `cover_image` string nullable
- `is_top` boolean default false
- `is_featured` boolean default false
- `price_gold` int default 0
- `view_count` int default 0
- `review_count` int default 0
- `last_review_at` timestamp nullable
- `published_at` timestamp nullable
- `deleted_at` softDeletes
- timestamps

**`article_categories` 表**：
- `id` ulid
- `parent_id` ulid nullable（自引用）
- `name` string
- `slug` string unique
- `sort_order` int default 0
- timestamps

**`article_tags` 表**：
- `id` bigIncrements
- `name` string
- `slug` string unique
- `article_count` int default 0
- timestamps

**`article_tag_pivot` 表**：
- `article_id` ulid
- `tag_id` unsigned bigint
- 复合主键

**`article_snapshots` 表**（暂不写逻辑，先建表）：
- `id` ulid
- `article_id` ulid
- `review_round` int
- `content_snapshot` longText
- timestamps

#### 1.2 Model

- `Article`：`HasFactory, SoftDeletes`，`user()`, `category()`, `tags()` 关联，`scopePublished()`
- `ArticleCategory`：`parent()`, `children()` 自引用
- `ArticleTag`：`articles()` 关联
- `ArticleSnapshot`：`article()` 关联

**注意 ULID**：用 `Illuminate\Database\Eloquent\Concerns\HasUlids` trait。

#### 1.3 Markdown 渲染

- [ ] 安装 `league/commonmark`（轻量、纯 PHP）
- [ ] `app/Services/MarkdownRenderer.php`：封装渲染方法，返回 HTML
- [ ] 服务端渲染存到 `content_html`？**不存** —— 接口渲染时实时算，存原文 + 渲染 + HTML Purifier
- [ ] 安装 `ezyang/htmlpurifier`
- [ ] 渲染流程：Markdown → HTML → Purifier 清洗 → 返回

#### 1.4 控制器 + 路由

**`ArticleController`**：

| 方法 | 路由 | 鉴权 | 逻辑 |
|---|---|---|---|
| `index` | `GET /api/articles` | 可选 | 分页 + 筛选（category_id, tag, author_id, status, q, sort） |
| `show` | `GET /api/articles/{id}` | 可选 | 详情 + 渲染 + 付费判断 |
| `store` | `POST /api/articles` | 需要 | 创建 |
| `update` | `PUT /api/articles/{id}` | 本人/管理员 | 更新 |
| `destroy` | `DELETE /api/articles/{id}` | 本人/管理员 | 仅 draft 可删 |
| `submit` | `POST /api/articles/{id}/submit` | 本人 | draft → pending |

**关键点**：
- `index` 默认 `status=published`，付费文章未登录/未付费只返回标题+封面
- `show` 用 MarkdownRenderer 实时渲染
- `store/update` 中 `tags` 数组：先 upsert 到 `article_tags`，再 attach 到 pivot
- slug 自动生成：`Str::slug($title)` + 随机后缀防冲突
- `update` 中 `published` → `pending` 的逻辑先实现，状态机放 M2

#### 1.5 资源类

- [ ] `ArticleListResource`：列表用的精简结构
- [ ] `ArticleDetailResource`：详情用的完整结构（含 author, category, tags, content, content_html）
- [ ] 付费文章未付费时省略 `content`/`content_html`

#### 1.6 分类/标签的简单接口（暂时）

- [ ] `GET /api/categories` 返回树形
- [ ] `GET /api/tags?q=xxx` 返回列表

> 完整的管理接口放 M3

### 验证方式

```bash
# 创建文章
curl -X POST http://127.0.0.1:8000/api/articles \
  -H "Authorization: Bearer <token>" \
  -H "Content-Type: application/json" \
  -d '{"title":"测试文章","content":"# 标题\n\n正文","tags":["PHP","Laravel"]}'
# 期望：201 + 文章对象

# 列表
curl "http://127.0.0.1:8000/api/articles?page=1&per_page=20"
# 期望：200 + data + meta
```

## M2 — 审核流程（P0-3）

### 目标
- 状态机跑通
- 机审逻辑嵌入 `submit`/`resubmit`
- 3 个用户端审核接口可用

### 任务清单

#### 2.1 状态机服务

- [ ] `app/Services/ArticleStateMachine.php`
- [ ] 用 PHP 8.2+ enum 定义 9 个状态
- [ ] 方法：`canTransition(from, to, action)`，返回 bool
- [ ] 状态转移图：

```
draft ──submit──► pending ──first_pass──► published
                      │                        │
                      │ first_reject            │ update
                      ▼                        ▼
                first_reject ◄──modify_required
                      │                        │
                      │ appeal                 │ resubmit
                      ▼                        ▼
                 appealing ──second_pass──► published
                      │
                      │ second_reject
                      ▼
                 second_reject (终态)
```

#### 2.2 机审服务

- [ ] `app/Services/SensitiveWordChecker.php`
- [ ] 方法：`check(string $content): array`，返回 `['level' => 'light|medium|heavy|none', 'matches' => [...]]`
- [ ] MVP 先用最简单的字符串匹配：`str_contains` + 关键词数组（从 config 读）
- [ ] 真正的 CRUD 放 M3，这里先用 config/seed 灌入测试词

#### 2.3 审核动作逻辑

修改 `ArticleController::submit`：
1. 调用 `SensitiveWordChecker::check`
2. `light`：替换为 `***` 后进入 `pending`，记一条 `auto_reject`-with-light 的 log（或者直接进 pending，记一条 info log）
3. `medium` / `heavy`：直接 `first_reject`，记 log
4. 干净：进 `pending`

#### 2.4 审核日志

- [ ] `article_review_logs` 迁移（按 `数据库设计.md`）
- [ ] `ArticleReviewLog` Model
- [ ] 每个状态转移都记一条
- [ ] `action` 枚举：`auto_reject` / `first_pass` / `first_reject` / `modify_required` / `second_pass` / `second_reject`

#### 2.5 三个用户端接口

| 接口 | 逻辑 |
|---|---|
| `GET /api/articles/{id}/review-logs` | 仅作者/管理员可看 |
| `POST /api/articles/{id}/appeal` | `first_reject` → `appealing`，写日志 |
| `POST /api/articles/{id}/resubmit` | `modify_required` → 走机审 → `pending`，写日志 + `review_count++` |

#### 2.6 快照（仅 `modify_required` 时）

- [ ] `submit` 时检查：若文章已有 `review_count > 0`，且当前是 `modify_required`，**先写快照再改**
- [ ] 快照逻辑封装在 `ArticleSnapshotService::saveSnapshot($article)`

### 验证方式

```bash
# 提交一篇文章触发机审
curl -X POST http://127.0.0.1:8000/api/articles/{id}/submit \
  -H "Authorization: Bearer <token>"
# 期望：200，状态变成 pending 或 first_reject（取决于内容）

# 看审核日志
curl http://127.0.0.1:8000/api/articles/{id}/review-logs \
  -H "Authorization: Bearer <token>"
# 期望：data 数组
```

跑通后开前端 M2 对话。

```php
class EnsureAdmin
{
    public function handle(Request $request, Closure $next)
    {
        if (Auth::user()?->role !== 'admin') {
            return response()->json(['message' => 'Forbidden'], 403);
        }
        return $next($request);
    }
}
```

- [ ] 这是给管理员用的，与 H5 工作台功能重复，**二选一**：
  - 优先做 H5 工作台（统一体验）
  - Filament 作为补充（数据看板、批量操作）

**建议**：本阶段只做 H5 工作台 API，Filament 留到后续或不做。

#### 5.1 关注

- [ ] `follows` 表：
  - `id` ulid

- [ ] `FollowController`：`POST /api/users/{id}/follow`、`DELETE /api/users/{id}/follow`

- [ ] `comments` 表（按 `数据库设计.md`）
- [ ] `comment_likes` 表
- [ ] `CommentController`：
  - `GET /api/articles/{id}/comments`：树形（一级 + 二级嵌套）

- [ ] 评论点赞：`POST /api/comments/{id}/like`（轻量，鉴权即可）

#### 5.3 金币 + 硬币

**`gold_transactions` 表**（充值货币流水）：
- `id` ulid

**每日硬币**：

- [ ] 检查 `last_coin_at` 是否今天，是则跳过；否则 `coin_balance++` + 写流水 + 更新 `last_coin_at`

**每日投币上限 5 枚**：在 `POST /api/articles/{id}/coin` 接口里检查今日已投出数（`coin_transactions` 聚合）。

#### 5.4 打赏 + 付费阅读

- [ ] `article_purchases` 表：
  - `id` ulid
  - `user_id` ulid
  - `article_id` ulid
  - `gold_amount` int
  - `purchased_at` timestamp
  - 唯一 `(user_id, article_id)`
- [ ] `POST /api/articles/{id}/reward`：扣打赏者金币 + 给作者 70% + 平台 30%
- [ ] `POST /api/articles/{id}/purchase`：扣读者金币 + 给作者 70% + 平台 30% + 写 `article_purchases`
- [ ] **充值接口**：`POST /api/wallet/recharge`，MVP 直接加金币，不接真实支付
- [ ] **下架退款**：放后续迭代，M5 不做

#### 5.5 触发通知

- 评论文章 → 给作者发通知
- 被打赏 → 给作者发通知
- 被付费阅读 → 给作者发通知
- 被关注（特殊关注） → 给被关注者发通知

### 验证方式


### 错误处理约定

- 业务错误：返回 `{ "message": "...", "errors": {...} }`，HTTP 400/403/422
- 找不到资源：404 + `{ "message": "Resource not found" }`
- 未认证：401
- 服务器错误：500 + 不暴露细节（生产环境）

### 验证规则（FormRequest）

每个写操作都建一个 FormRequest：
- `RegisterRequest`
- `LoginRequest`
- `StoreArticleRequest`
- `UpdateArticleRequest`
- `ReviewArticleRequest`
- 等等

### 测试

每个阶段至少跑通 curl 验证脚本。M3 之后可以加 PHPUnit Feature Test，但 MVP 不强制。

### 性能注意

- 文章列表：必须 `with(['user', 'category', 'tags'])` 预加载，避免 N+1
- 详情接口：渲染 Markdown 是 CPU 密集操作，可以加缓存（`Cache::remember`）

## 阶段性产物清单

每个 M 跑通后应该有以下可运行产物：

- [ ] M0：注册 → 登录 → 拿 token → 看到自己

- [ ] M2：提交 → 审核 → 看日志
- [ ] M3：管理员审 → 用户看到结果
- [ ] M4：审核后用户收到通知
- [ ] M5：关注 → 评论 → 打赏 → 付费 → 完整 MVP
