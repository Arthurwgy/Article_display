## M0 — 项目骨架 + 认证（2026-06-25）

### D1. `users.id` 使用 ULID（26 位字符串）

- **倾向**：从 M0 起步就用 `Illuminate\Database\Eloquent\Concerns\HasUlids` trait，主键为 26 位 ULID
- **理由**：
  - 接口文档「通用规范」明文规定 ID 类型为「字符串 ULID（26 位）」
  - M1 起的所有表都明确用 `HasUlids`，M0 起步统一省事
  - register / me 响应里的 `id` 字段直接返回字符串，零转换
- **影响文件**：`database/migrations/*_create_users_table.php`、`app/Models/User.php`

---

### D2. `coin_balance` / `gold_balance` 字段放在 `users` 表（M0 阶段）

- **倾向**：M0 在 `users` 表加 `coin_balance`（int default 0）和 `gold_balance`（decimal(10,2) default 0），**不建** `user_wallets` 表
- **理由**：
  - 执行方案 0.2 明确要求在 users 表加这两个字段
  - 接口文档的 register / me 响应里需要直接返回这两个值，留在 users 表少一次 join
  - 数据库设计文档 3.1 的「后续迁到 user_wallets」属于「规范建议」，M0 先按执行方案走；如后续要迁移，在 M5 钱包阶段再做
- **影响文件**：`database/migrations/*_create_users_table.php`、`app/Models/User.php`（casts）
- **后续关注**：M5 涉及金币/硬币事务时，如果需要行级锁或事务一致性，再考虑迁到 `user_wallets` 表

---

### D3. `users.name` 字段加唯一校验

- **倾向**：register 接口加 `unique:users,name` 校验
- **理由**：数据库设计文档 3.1 明文规定 `name string(50) 唯一`；接口文档未列此校验是疏漏，应以 DB 设计文档为准
- **影响文件**：`app/Http/Requests/RegisterRequest.php`（或 AuthController::register 内联 Validator）
- **错误提示**：「该昵称已被使用」

---

### D4. 迁移加 `email_verified_at` 和 `remember_token` 字段（M0 不写验证流程）

- **倾向**：迁移文件加这两个字段，但 M0 不实现邮箱验证、不启用「记住我」
- **理由**：
  - 数据库设计文档 3.1 表结构里包含这两个字段
  - Laravel 默认 `users` 表迁移也带这俩，与框架惯例对齐
  - M0 阶段用不到，但后续如要做邮箱验证或记住登录可省一次迁移
- **影响文件**：`database/migrations/*_create_users_table.php`

---

### D5. CORS `allowed_origins` 加 `localhost:9000`

- **倾向**：CORS 白名单包含 `http://localhost:5173`、`http://localhost:8080`、`http://localhost:9000`
- **理由**：
  - `5173` 是 Vite 默认端口
  - `8080` 和 `9000` 都是 UniApp H5 开发常用端口（`npm run dev:h5` 默认 8080，部分 manifest 配置可改 9000）
  - 多加一个避免前端换端口时后端报错
- **影响文件**：`config/cors.php`

---

### D6. 管理员账号为 `admin@example.com` / `password`

- **倾向**：AdminSeeder 创建 `admin@example.com`，密码 `password`（bcrypt），role=admin
- **理由**：执行方案 0.5 明确指定
- **影响文件**：`database/seeders/AdminSeeder.php`
- **仅限开发环境**：此账号与密码为开发占位，**严禁**在生产环境使用

## M1 — 文章 CRUD（2026-06-25）

### D7. `articles.status` enum 值以执行方案为准

- **倾向**：`draft` / `pending` / `first_pass` / `published` / `first_reject` / `modify_required` / `appealing` / `second_pass` / `second_reject`
- **理由**：接口文档 P0-2 明确规定；现有迁移里用的是另一套命名（`pending_approved`/`rejected` 等），以接口文档为准
- **影响文件**：`database/migrations/2026_06_25_000003_create_articles_table.php`
- **上游冲突**：数据库设计文档中 `articles` 表的 status enum 描述与接口文档不一致，以接口文档为准

---

### D8. 所有涉及 users 的外键统一用 ULID 类型

- **倾向**：所有引用 `users.id` 的外键列（`user_id`、`follower_id`、`following_id` 等）统一用 `$table->ulid()` + 外键约束
- **理由**：users 主键已改为 ULID，`foreignId()` 生成 `bigint`，外键类型必须匹配 MySQL 才能建约束
- **影响文件**：所有引用 users 的迁移（M1–M5 共约 12 个迁移文件）
- **覆盖范围**：`articles.user_id`、`follows.follower_id/following_id`、`notifications.user_id`、`gold_transactions.from/to_user_id`、`coin_transactions.user_id`、`article_paid_access.user_id`、`comment_likes.user_id`、`comments.user_id`

---

### D9. `league/commonmark` v2.8 用 `CommonMarkConverter` API

- **倾向**：用 `CommonMarkConverter`（已内置 CommonMark GFM 支持），`convert()` 方法返回 `Stringable`
- **理由**：`MarkdownRenderer` 类名与 `League\CommonMark\MarkdownRenderer` 冲突；`Environment::merge()` 在 v2 中已不存在；`MarkdownRenderer` 服务类需用别名 `as CommonMarkRenderer` 避免冲突
- **影响文件**：`app/Services/MarkdownRenderer.php`
- **渲染流程**：Markdown → `CommonMarkConverter::convert()` → `HTMLPurifier::purify()` → 返 HTML

---

### D10. `published` 状态时自动填 `published_at`

- **倾向**：创建文章时若 `status = published`，自动填 `published_at = now()`
- **理由**：接口文档响应要求 `published_at` 字段；已发布文章列表按 `published_at` 排序
- **影响文件**：`app/Http/Controllers/Api/ArticleController.php::store()`

---

### D11. `article_categories.id` 使用 ULID（bigIncrements 改 ULID）

- **倾向**：分类表主键从 `$table->id()`（bigint）改为 `$table->ulid('id')->primary()`
- **理由**：`articles.category_id` 已改为 ULID，类型必须匹配才能建外键约束
- **影响文件**：`database/migrations/2026_06_25_000002_create_article_categories_table.php`、`app/Models/ArticleCategory.php`（加 `HasUlids` trait）

### D12. `article_review_logs.action` enum 值以迁移规范为准

- **倾向**：`submit` / `auto_reject` / `first_pass` / `first_reject` / `modify_required` / `appeal` / `second_pass` / `second_reject`

- **影响文件**：`database/migrations/2026_06_25_000006_create_article_review_logs_table.php`

### D13. `SensitiveWordChecker` config 中文 key 编码问题

- **问题**：`config/sensitive_words.php` 中的中文 key（如 `'违禁'`、`'敏感'`）在 Laravel 加载时被错误解析为乱码，导致所有敏感词匹配失效
- **根因**：`bootstrap/cache/config.php` 缓存了旧编码的 config，Laravel 在 `APP_ENV=local` 下不自动重新编译；加上 Windows 文件写入编码问题
- **临时方案**：config 中的词 key 全部改为英文（如 `'guilty'`、`'illegal'`），中文词仅作注释说明
- **后续**：M3 接入 `SensitiveWord` 数据库表后，词库从 DB 读取，此 config 文件改为从 DB 读取的 Service，届时不再受 config 编码问题影响
- **影响文件**：`config/sensitive_words.php`、`app/Services/SensitiveWordChecker.php`
- **附加修复**：`SensitiveWordChecker::mask()` 中 `mb_str_replace` 不是 PHP 标准函数，改为 `str_replace`

---

### D14. `ArticleReviewLog` Model 缺少 `HasUlids` trait

- **问题**：提交文章时报 `SQLSTATE[HY000]: General error: 1364 Field 'id' doesn't have a default value`，日志表插入失败
- **根因**：`ArticleReviewLog` Model 未引入 `HasUlids` trait，ULID 主键无法自动生成
- **修复**：Model 中添加 `use HasUlids;`
- **影响文件**：`app/Models/ArticleReviewLog.php`

---

### D15. `ArticleSnapshotService::saveSnapshot` 漏填 `title_snapshot` 字段

- **问题**：`modify_required` 重提交时快照写入报错 `Field 'title_snapshot' doesn't have a default value`
- **根因**：快照表迁移含 `title_snapshot`/`cover_image_snapshot` 字段，但 `saveSnapshot` 只填了 `content_snapshot`
- **修复**：`updateOrCreate` 时补填 `title_snapshot` 和 `cover_image_snapshot`
- **影响文件**：`app/Services/ArticleSnapshotService.php`
