## M0 — 项目骨架 + 认证（2026-06-25）

### D1. 清理 Hello uni-app 模板的 4 个 tabBar 页面 + 配套图标 + 窗口

- **倾向**：M0 阶段把 Hello 模板自带的 4 个 tabBar 页面（`pages/tabBar/{component,API,template,extUI}/`）**连同 4 对 tabBar 图标**（`static/{component,api,extui,template}.png` 及对应 `*HL.png`）一起从磁盘删除，并从 `pages.json` 移除 4 条 tabBar 配置；`pages.json` 数组中其他 30+ 个非 tabBar 演示页面（`pages/component/...`、`pages/API/...`、`pages/extUI/...`、`pages/template/...`、`platforms/...`、`pages/about/about`）同步删除
- **理由**：
  - 执行方案 0.7 明文要求「先删掉 Hello uni-app 自带的 4 个 tabBar 页面，避免误导」
  - 但仅删 4 个 tab 页面而不删配套图标会造成 `static/` 下 8 个孤儿文件；不删其他演示页面会污染 `pages.json` 路由表，后续业务页面会混入大量 Hello 模板路径
  - 一次性清理到位，避免后续阶段反复打补丁
- **影响文件**：
  - 删除：`apps/pages/tabBar/`、`apps/pages/component/`、`apps/pages/API/`、`apps/pages/extUI/`、`apps/pages/template/`、`apps/pages/about/`、`apps/pages/error/`、`apps/platforms/`、`apps/windows/`（如存在）
  - 删除：`apps/static/{component,api,extui,template}.png` 及 `*HL.png` 共 8 个文件
  - 改：`apps/pages.json`（清空 pages 数组、tabBar 数组，新加 4 个业务页面）
- **保留**：`apps/store/index.js`（Vuex）、`apps/store/counter.js`、`apps/common/uni.css`、`apps/common/uni-nvue.css`（详见 D2）
- **后续关注**：保留的 `common/uni.css` 是 Hello 模板的全局 reset，M0 不依赖但留着不影响；M5 收尾阶段如需彻底瘦身可再删

### D2. 保留工程内 Vuex 主 store + counter 模块，Pinia 作为新增业务状态层

- **倾向**：M0 **不动** `apps/store/index.js`（Vuex）和 `apps/store/counter.js`；新建 `apps/store/modules/user.js` 用 Pinia 的 `defineStore`，与 Vuex 并存
- **理由**：
  - `apps/main.js` VUE3 分支已存在 `app.use(store)`（Vuex）和 `app.use(Pinia.createPinia())` 两行并存，框架本身支持双状态
  - 模板里的 Vuex 已被 `app.use(store)` 全局注入，删除会引入不必要的回归风险
  - 业务代码（M0 的 user store、未来 M4 的 notification store）统一走 Pinia，Vuex 留作"不写不读"的死代码
  - 一次到位拆 Vuex 会涉及 `main.js`、`App.vue`（用了 `mapMutations`）等多处修改，**不属于 M0 任务清单**，放到 M5 收尾阶段统一处理
- **影响文件**：
  - 新增：`apps/store/modules/user.js`
  - 不动：`apps/main.js`、`apps/store/index.js`、`apps/store/counter.js`、`apps/App.vue`（其 `mapMutations` 是给 Vuex 用的，M0 不调用）
- **后续关注**：M5 收尾时统一删 Vuex，包括从 `main.js` 移除 `app.use(store)`、删 `store/index.js` 和 `store/counter.js`、清理 `App.vue` 的 `mapMutations` 引用

### D3. `utils/index.js` 只做基础工具聚合，不导出 API / Store

- **倾向**：`apps/utils/index.js` 导出 `request`（HTTP 封装）和 `auth`（token 工具）两个具名导出；**不导出** `api/*`、`store/modules/*`
- **理由**：
  - API 模块走 `@/api/auth` 这类显式路径，便于 IDE 跳转和 tree-shaking
  - Store 模块走 `useUserStore()` 函数调用，不适合二次封装导出
  - 聚合文件只为 `import { request } from '@/utils'` 这种基础用法服务
- **影响文件**：`apps/utils/index.js`
- **API 引用约定**：
  - 业务页：`import { authApi } from '@/api/auth'`
  - 用户状态：`import { useUserStore } from '@/store/modules/user'`
  - HTTP：`import { request } from '@/utils'` 或 `import { request } from '@/utils/index'`

### D4. tabBar 用文字模式（不配 iconPath），H5 端纯文字按钮

- **倾向**：`pages.json` 的 tabBar `list` 数组里**只填 `pagePath` + `text`**，不填 `iconPath` / `selectedIconPath`
- **理由**：
  - M0 是骨架阶段，2 个 tab（首页 / 我的）用纯文字足够辨识
  - 不配图标可以规避 8 个孤儿图标删除后 iconPath 引用悬空的问题
  - 不为 M0 临时画图标，把"统一图标方案"留到 M5 收尾阶段（届时需要 4-6 个图标：首页、我的、发布、消息、钱包等）
- **影响文件**：`apps/pages.json`（tabBar 配置）
- **后续关注**：M5 收尾阶段统一设计/采购一套 tabBar 图标

### D5. 401 拦截在 `request.js` 内部完成，不依赖业务页 onShow 检测

- **倾向**：`apps/utils/request.js` 拦截 `statusCode === 401` 时**主动**清 Pinia token + 清 storage + `uni.reLaunch('/pages/auth/login')`；mine 页 onShow 仍按文档做"无 token 跳登录"作为兜底
- **理由**：
  - 把 401 集中处理在请求层，避免每个业务页重复写「token 失效就跳登录」
  - 文档 0.1 明确要求"统一处理 401（清 token + 跳登录）"，这里把行为固定为**自动跳转 + toast 提示**
  - mine onShow 的兜底仍保留，覆盖"本地有 token 但后端已失效"和"首次进 mine 还没调 fetchMe"两种场景
- **影响文件**：
  - `apps/utils/request.js`：401 分支调用 `useUserStore().clear()`（Pinia action）+ `uni.removeStorageSync('token')` + `uni.reLaunch`
  - `apps/store/modules/user.js`：新增 `clear()` action（清 token + userInfo + storage）
- **注意事项**：
  - 401 跳登录时**不**弹 toast（避免和登录页初始化时 fetchMe 失败弹的 toast 重复）
  - 跳登录的 URL 走常量 `LOGIN_PATH = '/pages/auth/login'`，避免硬编码

### D6. 注册成功 = 写 token + userInfo + switchTab 到 mine，不调 login

- **倾向**：注册接口响应里已带 `token` + `user`，`userStore.register()` action **直接消费响应**，写 Pinia + storage，**不**再调一次 `authApi.login()`；跳转用 `uni.switchTab({ url: '/pages/mine/mine' })`
- **理由**：
  - 接口文档 1.1 register 响应结构与 login 一致（`{ user, token, token_type, expires_in }`），重复调 login 是浪费
  - `uni.switchTab` 是 tabBar 页面标准跳转，H5 下行为等同 redirectTo
  - 文档 0.4 「成功后自动登录」按字面理解就是「不再走 login 流程」，用 switchTab 一次性落到 mine
- **影响文件**：
  - `apps/store/modules/user.js`：`register` action
  - `apps/pages/auth/register.vue`：`register` 成功后 `switchTab`
- **失败回滚**：
  - register 422（参数错误 / 昵称已被使用）→ toast 显示 `errors` 第一条，**不**写 token
  - register 5xx → toast 通用错误，**不**写 token
  - register 成功但 switchTab 失败（H5 极少见）→ 兜底走 `uni.reLaunch`

### D7. mine 页显示首字母圆形占位头像（`avatar: null` 时）

- **倾向**：`pages/mine/mine.vue` 渲染头像时，先判断 `userInfo.avatar`；为空时显示**昵称首字符的圆形占位**（背景色按 `userInfo.id` 哈希取 6 种固定色之一，保证同用户颜色稳定）
- **理由**：
  - 接口文档 1.1 / 1.4 register 和 me 响应里 `avatar` 字段就是 `null`（默认无头像）
  - M0 没有上传头像功能（P5 都不一定有真实上传，"不防爬"原则下头像可选）
  - 首字母占位是最轻的视觉兜底，比显示"暂无头像"文字好看
- **影响文件**：
  - `apps/pages/mine/mine.vue`：渲染逻辑 + 一个内置的占位头像子组件或纯 CSS 圆形
  - 不新增独立组件（避免 M0 多文件）
- **颜色算法**（内置，不做工具函数）：
  ```js
  const colors = ['#5B8FF9', '#5AD8A6', '#F6BD16', '#E8684A', '#6DC8EC', '#9270CA']
  const idx = (userInfo.id || 'x').charCodeAt(0) % colors.length
  const bg = colors[idx]
  ```
- **后续关注**：M3 / M5 阶段如引入头像上传功能，本兜底逻辑自然降级为 fallback

### D8. mine 页"未登录态跳登录"用 `uni.reLaunch` 而不是 `redirectTo`

- **倾向**：`pages/mine/mine.vue` 在 `onShow` 里检测 `!token` 时调 `uni.reLaunch({ url: '/pages/auth/login' })`，不调 `redirectTo` 也不调 `navigateTo`
- **理由**：
  - mine 是 tabBar 页面，tabBar 页面用 `navigateTo` 会报错（"无法通过 navigateTo 跳转 tabBar 页面"），用 `switchTab` 跳 login 也错（login 不是 tabBar）
  - `reLaunch` 能清空页面栈，避免用户登录后按返回又回到 mine 形成循环
  - 文档 0.5 验收项第 5 条「退出登录 → 回到我的页时跳登录页」对应的就是这个 reLaunch 行为
- **影响文件**：`apps/pages/mine/mine.vue`（`onShow` 检测 + reLaunch）
- **依赖**：D5 的 401 拦截也用 `reLaunch`，保持跳转 API 一致

### D9. forgot-password.vue 创建为占位页（含返回按钮），不只放静态文案

  - 有"返回登录"按钮比纯静态文案更符合用户预期
  - 文件 < 30 行，不增加维护成本

### D10. 注册 / 登录页显式渲染 422 验证错误（`errors` 字段）

- **倾向**：`apps/pages/auth/register.vue` 和 `login.vue` 拿到 422 响应时：
  - 优先取 `res.data.errors` 第一个字段的第一条 message
  - 没 `errors` 时回退到 `res.data.message`
  - 都没时显示 "请求失败"
- **理由**：
  - 后端 D3 决定 register 加 `unique:users,name` 校验，422 响应会是 `{ message: "...", errors: { name: ["该昵称已被使用"], email: ["..."] } }`
  - 文档 0.4 注册页要求"校验两次密码一致"，这是**前端**校验；后端 422 也得友好显示
  - 不在每个字段下面显示红字（M0 表单简单，统一 toast 即可）
- **影响文件**：
  - `apps/pages/auth/register.vue`
  - `apps/pages/auth/login.vue`
- **request.js 配合**：422 不走 toast，由业务页自行处理（request.js 在 422 时**只 reject，不 toast**）
