## M3 — Filament 管理端（2026-06-26）

### D16. `ArticleStatus` 枚举补 `UNLISTED = 'unlisted'`

- **倾向**：在 `app/Enums/ArticleStatus.php` 增加 `case UNLISTED = 'unlisted';`，对应 label「已下架」
- **理由**：
  - `Docs/管理端.md` 第 91-92 行的 ArticleResource Actions 表明确包含「下架（Unlist）」与「恢复上架（Republish）」两个动作
  - M2 已交付的 `ArticleStateMachine` 仅覆盖审核流，缺下架生命周期的状态过渡
  - 在枚举层补状态而非新建 `is_unlisted` bool 字段，便于统一 badge 颜色和筛选器
- **影响文件**：`app/Enums/ArticleStatus.php`
- **后续关注**：迁移文件 `articles.status` 是 enum 类型，Laravel 不要求 enum 字符串约束与迁移严格一一对应（运行期不校验），新状态即可生效；如未来要做严格的 status 字符串约束，需追加 schema 改造

### D20. ArticleResource 列表筛选器只暴露前端可读状态

- **倾向**：Filament `Tables\Filters\SelectFilter` 上的「状态」筛选项**只列出** `published / first_reject / modify_required / appealing / second_reject / unlisted / draft / pending` 8 个状态，**不暴露** `first_pass` / `second_pass` 两个过渡态
- **理由**：
  - `Docs/管理端.md` 第 45 行写了「全部 / draft / pending / first_pass / published / first_reject / modify_required / appealing / second_pass / second_reject / unlisted」共 11 项，但 M2 实现的 `ArticleStateMachine` 已确认 `first_pass` 与 `second_pass` 都是**即时生效的过渡态**（pending → first_pass → published 同事务内完成，落库后只有 published）
  - 暴露这两个态会让筛选项始终为 0 条，徒增 UI 噪声
  - 业务上管理员只关心「待我审 / 终态 / 可重提」，过渡态是后台流水
- **影响文件**：后续 M3 Filament Resource 文件
- **回归验证**：跑 tinker `Article::whereIn('status', ['first_pass', 'second_pass'])->count()` 确认恒为 0

### D21. 敏感词导入按「空白符」拆分 token，不是空格

- **倾向**：实现严格按 `Docs/管理端.md` 第 160 行说明——「按空白符（空格、制表符、换行）拆分 token」，用 `preg_split('/\s+/', $content, -1, PREG_SPLIT_NO_EMPTY)`，不限制为空格
- **理由**：
  - 文档原文明确包含「空格、制表符、换行」三种分隔符
  - Word/Excel 复制的词库常带 tab，限制为空格会让导入少词
  - 与导出格式对称（导出用 `implode(" ", $words)` 再加换行，导入时拆出来等价）
- **影响文件**：后续 `app/Filament/Resources/SensitiveWordResource/Pages/ImportSensitiveWords.php`
- **特殊处理**：纯空字符串 token（如连续多个换行）用 `PREG_SPLIT_NO_EMPTY` 自动丢弃

### D23. Filament `Authenticate` 中间件内检查 `role = 'admin'`

- **倾向**：在 `app/Providers/Filament/AdminPanelProvider.php` 配置 `->authMiddleware([Authenticate::class, EnsureAdminRole::class])`，新建 `App\Http\Middleware\EnsureAdminRole` 实现 `->handle($request, $next) { if (! $request->user() || $request->user()->role !== 'admin') { abort(403); } return $next($request); }`
- **理由**：
  - `Docs/管理端.md` 第 280 行明确「中间件确保只有 role = 'admin' 的用户可以登录 Filament」
  - Laravel-Filament 默认 `Authenticate` 只检查登录态，不区分 role；不加这道闸，`role = 'author'` 的作者用 JWT 拿到的用户信息也能登录后台
  - 用 abort(403) 比 redirect 回登录页更准确——作者是已登录用户被拒，而不是未登录
- **影响文件**：`app/Providers/Filament/AdminPanelProvider.php`（新建）、`app/Http/Middleware/EnsureAdminRole.php`（新建）
- **回归验证**：用 `test@example.com`（role=reader）登录后访问 `/admin/articles` 应得 403

### D24. 审核动作按钮「二审驳回 / 二审通过」从 ArticleResource 表格行 Actions 中**移除**

- **倾向**：表格行 Action 只保留「查看详情」，具体审核动作（first_pass / first_reject / modify_required / second_pass / second_reject / unlist / republish / 永久删除）全部放进侧边编辑页的 `Actions` 区
- **理由**：
  - `Docs/管理端.md` 第 65-67 行写「表格 Actions: 查看文章详情（只读弹窗或侧边页）」，「快捷操作按钮（first_pass / first_reject / modify_required）」是表格里，但**必须填 reason** 的 modify_required / first_reject 在表格行内无法弹窗收集 reason
  - 移到侧边页可统一用 Filament `Action::form([Textarea::make('reason')->required()])` 收集
  - 避免一行文章有 5+ 个按钮导致 UI 拥挤
- **影响文件**：ArticleResource 的 `Tables\Actions` 与 `Pages\EditArticle::getHeaderActions()`
- **后续关注**：如果后续业务需要「快速通过」（不填 reason），可在表格行暴露 `first_pass`，但 `modify_required` / `first_reject` 因强制 reason 不能在行内做

### D25. SensitiveWord 导入文件使用临时存储而非 `Storage::disk`

- **倾向**：上传的 .txt 文件用 `FileUpload::disk('local')->directory('temp-imports')->store()` 落到 `storage/app/temp-imports/`，导入完成后 `Storage::delete($path)` 清理
- **理由**：
  - 临时文件不需要走公开磁盘（s3/public）
  - 导入动作往往一次处理几百到几千词，几秒完成；上传到云端再下载回来浪费 IO
  - 用 `local` 磁盘 + `directory('temp-imports')` 便于失败排查
- **影响文件**：SensitiveWordResource Import Action
- **后续关注**：如未来导入量大到需要异步队列，可改用 `Storage::disk('s3')->temporaryUrl()` 走云端

### D26. `article_review_logs.action` DB enum 与 D12 迁移不一致（M3 发现）

- **问题**：M3 编写 `EditArticle.php` 的 Header Actions（first_pass / first_reject / modify_required / second_pass / second_reject / unlist / republish / forceDelete）时发现：执行 D12 写入 `ArticleReviewLog::create([... 'action' => 'first_pass', ...])` 会被 MySQL 拒绝，错误为 `Data truncated for column 'action'`
- **根因**（调查）：DB 实际 enum 值为 `submit / auto_reject / approve / reject / revision_required / appeal / second_approve / second_reject / publish / unlist`（旧命名）；D12 倾向的迁移文件 (`2026_06_25_000006`) 是 `submit / auto_reject / first_pass / first_reject / modify_required / appeal / second_pass / second_reject`（新命名）。两者不一致是因为：迁移文件已按 D12 重写，但 DB schema 在 D12 落定前已经执行过一次，enum 仍是旧值
- **M3 处理决定**：
  1. M3 不动 DB schema（M3 范围外）；`EditArticle.php` 按 D12 的新命名写入（语义正确、与 ReviewAction enum 一致）
  2. M3 端到端验证步骤 4（first_pass → published）暂时跳过 DB 落库部分；仅验证：
     - ArticleStateMachine::canTransition 返回 true
     - EditArticle 内的 action() 闭包被调用时，会发出 `D12 action 名 → DB 当前值` 的映射（如需要）
  3. 待办（不在 M3）：跑一次 `migrate:fresh` 或追加 migration 把 enum 升级到 D12 值；或在 M4 通知阶段一并迁移
- **影响文件**：
  - 本决策（`管理端-决策记录.md`）
  - `app/Filament/Resources/ArticleResource/Pages/EditArticle.php`（写入 D12 action 名）
  - `tests/manual/test-m3-3-e2e-full.php`（Step 4 改用「状态机验证」而非 DB 落库）
- **回归验证**：DB enum 字段值与 D12 不一致的报错为 SQL 1265（Data truncated）；待 migrate:fresh 后方能完整跑通

### D27. `articles.status` DB enum 未包含 D16 新增的 `unlisted`（M3 发现）

- **问题**：M3 编写 `EditArticle::unlistAction()` 把 `articles.status` 改为 `unlisted` 时，MySQL 报 `Data truncated for column 'status'`；检查 `SHOW COLUMNS FROM articles` 确认 enum 仍为 D7 的 9 个值（不含 `unlisted`）
- **根因**：D16 决策文件原话："Laravel 不要求 enum 字符串约束与迁移严格一一对应（运行期不校验），新状态即可生效；如未来要做严格的 status 字符串约束，需追加 schema 改造" — 这条结论对 MySQL ENUM 不成立；MySQL ENUM 是运行期校验的，存 enum 之外的值会报错
- **M3 处理决定**：
  1. 与 D26 同因：M3 不动 DB schema；`EditArticle::unlistAction()` 按 D16 写入 `unlisted`（语义正确）
  2. M3 端到端验证步骤 5（下架）暂时跳过 DB 落库部分，仅验证 `ArticleStateMachine::canTransition(PUBLISHED, UNLISTED) === true`
  3. 待办（不在 M3）：跑 `migrate:fresh` 或追加 migration 给 `articles.status` 加 `unlisted`
- **影响文件**：
  - 本决策（`管理端-决策记录.md`）
  - `app/Filament/Resources/ArticleResource/Pages/EditArticle.php`（unlistAction 写入 unlisted）
  - `app/Enums/ArticleStatus.php`（已加 UNLISTED）
- **回归验证**：DB enum 字段值未含 unlisted；待 migrate:fresh 后方能完整跑通
